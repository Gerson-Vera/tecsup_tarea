# ADR-001: Uso de Arquitectura Hexagonal (Ports & Adapters)

## Estado
Aceptado

## Fecha
2026-05-11

## Contexto
El sistema "Banco Digital" requiere separar claramente la lógica de negocio de los detalles técnicos (base de datos, API REST, notificaciones). Sin esta separación, cualquier cambio en la capa de persistencia o en el canal de notificación obliga a modificar el núcleo del sistema, aumentando el riesgo de regresiones y dificultando las pruebas unitarias.

## Decisión
Implementar Arquitectura Hexagonal (Ports & Adapters), organizando el código en tres capas: `domain`, `application` e `infrastructure`, donde el dominio no tiene ninguna dependencia externa.

## Alternativas Consideradas
1. **Arquitectura en capas tradicional (Layered)** — simple pero genera dependencias descendentes que acoplan dominio a infraestructura.
2. **Arquitectura Hexagonal** — ELEGIDA: invierte las dependencias y permite intercambiar adaptadores sin tocar el núcleo.
3. **Microservicios** — excesivo para el alcance del proyecto.

## Consecuencias

### Positivas
- El dominio (`Account`, `Transaction`) puede probarse sin Spring ni base de datos.
- Cambiar de H2 a MySQL solo requiere reemplazar el adaptador de persistencia.
- Agregar un canal de notificación (email, SMS) solo requiere una nueva implementación de `NotificationPort`.
- La regla crítica "domain NO depende de adapters" se cumple estructuralmente.

### Negativas
- Mayor cantidad de archivos y clases comparado con una arquitectura en capas simple.
- Requiere mappers entre entidades de dominio y entidades JPA.
